## Learned User Preferences

- Be skeptical and accurate over polite: steel-man a questionable request in one sentence, then say why it still fails; don't argue when the writing or code is already fine.
- Never invent or embellish project/resume details; ship placeholders rather than fabricated content.
- Do not use em-dashes or " -- " in prose written for the user; on the site, page/OG titles use the middot form ("Title · Niran D.") and date ranges use spaceless en dashes (e.g. "2022–Present", "2020–2021").
- Position the user as "AI-adjacent," never "AI-focused" (a deliberate choice to reduce interview risk); more broadly, for job-search copy the user wants to read as a versatile senior generalist, not a niche/field-specific expert, even if that means generalizing away from an over-used theme word (e.g. dropping a "trust"-heavy framing). Depth and specifics belong in the case studies, not the hero.
- Do not raise case-study attribution concerns for the user's projects (e.g. KAYAK, PartyPoint); the user has confirmed these are work they did.
- Portfolio font system (current): Playfair Display (serif, headings/display via the --display CSS var) + Source Sans 3 (body via --sans), a deliberate "classic serif + sans" pairing, both fetched live from Google Fonts (no self-hosted font files). Note: Source Sans Pro was renamed/delisted on Google Fonts in favor of Source Sans 3, which is the correct fetch target for that name going forward. This supersedes the earlier self-hosted Circular Std + Geist system, which itself had superseded an even earlier no-serif rule (Fraunces/Source Serif 4 were tried and dropped that time) — so "no serif" is no longer a standing constraint; go with whatever pairing the user asks for in the moment. Palette: bg #ffffff, ink rgba(0,0,0,.88), slate-blue accent #2f4a6b. Keep layouts clean and easy to read, with a left-aligned home hero and generous spacing on case studies.
- Wants tasteful delight/microinteractions on the portfolio (Web Audio click sounds, now always on with the header on/off toggle removed; a cursor "View Project (+)" label, magnetic CTAs, copy-email confirmation, avatar jump), but considers heavy effects (particle/comet cursor trails, image distortion, full-page parallax) too much for a hiring portfolio.
- When polishing portfolio copy, keep a credible senior product designer voice (clear, strategic, framed around problem, trade-offs, and impact), never salesy, dramatic, or influencer-y. Reduce AI-sounding language: avoid buzzwords (leveraged, seamless, delightful, innovative, robust, user-centric), "thesis/hook" style headlines, and "not X, but Y" antithesis/aphoristic openers, unless truly needed. Do not over-edit: only rewrite sections that need it and leave already-strong prose intact.
- Keep portfolio motion clean and restrained: the user explicitly rejects image zoom/Ken-Burns effects and shared-element image-morph transitions between the home and case-study pages as distractions. The motion they want is scroll-triggered text-highlight reveals (the highlight sweeps across the text left-to-right, firing when scrolled into view rather than on page load) and rolling-number counters for case-study stats. All motion is gated behind prefers-reduced-motion, and the Web Audio click sound is always on (the header on/off toggle was removed).
- Personal/side-project sections (e.g. the Photos section and the Zoom presentation-engagement app case study) should read as intentional product thinking that conveys personality (explores outside work, solves problems, brings joy), while staying restrained enough for a hiring portfolio.
- For the Photos section: avoid a plain gallery/grid and a map (the Instagram export's GPS is unreliable); organize by visual "modes of seeing," and do not show dates or a timeline (the user hasn't shot recently and does not want to signal recency).
- When the user shares a reference site they like (e.g. andreavollendorf.com for the home page, roupenjam.com for hero-copy structure), replicate its layout, typography, microinteractions, and even sentence structure (e.g. "[Role]. I [do X], [do Y], and [do Z].") faithfully rather than approximating, but never copy the literal wording; they push back hard when the result diverges from the reference. Case studies should be story-driven and visually compelling (a long page is fine) with interactive elements where they help, not walls of text, and outcomes stay honest and qualitative (no fabricated metrics).
